致命-1｜feedparserはタイムアウトを直接制御できない
対象：detail_design_v0.1.md セクション1-3（RSS Fetcher）
────────────────────────────────────────────────────────────────────────────────

【問題点】
セクション1-3のRSS Fetcherで feedparser.parse() にタイムアウトを設定しているが、
feedparserはタイムアウトパラメータを持たない。

  - feedparser.parse() は内部でurllibを使用しており、timeout パラメータを
    直接受け付けない
  - 基本設計で定義した「接続タイムアウト5秒・読み取りタイムアウト10秒」が
    実際には機能しない
  - タイムアウトなしでHoops Rumorsが無応答になった場合、フェールオーバーが
    発動せずバッチが無期限にハングする

【変更案】
feedparserに渡す前にrequestsでコンテンツを取得し、タイムアウトを制御する：

  def fetch_rss_source(source: dict) -> list:
      response = requests.get(
          source["url"],
          headers={"User-Agent": "NBANewsJP/1.0"},
          timeout=source["timeout"]   # (5, 10) が有効になる
      )
      response.raise_for_status()
      feed = feedparser.parse(response.content)
      return feed.entries

致命-2｜source_urlカラムが link と同値で冗長かつ設計に矛盾がある
対象：detail_design_v0.1.md セクション2-1（articlesテーブルDDL）
────────────────────────────────────────────────────────────────────────────────

【問題点】
セクション2-1のarticlesテーブルDDLで以下の記述がある：

  link        TEXT NOT NULL UNIQUE,  -- 重複排除キー
  source_url  TEXT NOT NULL,         -- 元記事URL（= link と同値）

コメントに「= link と同値」と明記されているが、同一データを2カラムに持つのは
設計上の問題。

  - link はUNIQUE制約で重複排除キーとして機能しており、source_url は不要な冗長カラム
  - 基本設計6-2のレスポンス例では source_url を返しているが、APIレスポンスでは
    link を source_url として返せばよい
  - 2カラムの値が常に同じである保証がなく、将来的なバグの温床になる

【変更案】
source_url カラムを削除し、APIレスポンスで link を source_url キーとして返す：

  # APIレスポンス生成時
  "source_url": article.link   # linkカラムの値をsource_urlとして返す

改善-1｜Claude APIのJSONDecodeError時の保存処理が設計と実装で食い違う
対象：detail_design_v0.1.md セクション1-2・1-5 / test_design_v0.1.md T-09
────────────────────────────────────────────────────────────────────────────────

【問題点】
セクション1-5のエラーハンドリング方針では「JSONDecodeError：当該記事をスキップ。
title_ja=title_original・summary_ja=NULLで保存」とある。しかし
process_article() の実装は return None を返すだけで、保存処理の制御は
呼び出し元（run_batch()）に委ねられている。

  - run_batch() の設計（セクション1-2）には process_article() が None を返した
    場合の保存処理が記述されていない
  - 「スキップして保存しない」のか「英語見出しのみで保存する」のか
    詳細設計上で一貫していない
  - T-09（API上限管理）のテストケース1では「process_article() が None を返す」
    ことを確認しているが、その後の保存動作を検証するテストが存在しない

【変更案】
run_batch() のセクション⑤の後に以下を明記する：

  ⑤ Claude API処理
     → None が返った場合（JSONエラー・バリデーション失敗）:
       title_ja = title_original（英語）
       summary_ja = NULL
       category = NULL
       has_score = False
       → DBに保存（⑦へ進む）
     → None が返った場合（RateLimitError）:
       api_limit_exceeded = "true" に更新済み
       → 当該記事をスキップ（DBに保存しない）

改善-2｜spurs_filter_enabled の設定変更がフロントエンド側でAPIに同期されていない
対象：detail_design_v0.1.md セクション3-1（useNewsフック）
────────────────────────────────────────────────────────────────────────────────

【問題点】
セクション3-1の useNews フックを確認すると、toggleSpoilerGuard() はAPI同期処理が
実装されているが、setSpursOnly() にはAPI同期処理がない。

  - spursOnly を切り替えてもバックエンドの app_settings.spurs_filter_enabled が
    更新されない
  - アプリ再起動後にフィルタ状態が初期値（false）に戻る
  - T-12（設定変更の即時反映）のケース1では spurs_filter_enabled の変更を確認
    しているが、この実装では変更がAPI同期されないためテストが通らない

【変更案】
setSpursOnly を toggleSpursFilter に変更し、API同期を追加する：

  const toggleSpursFilter = async () => {
      const next = !spursOnly;
      setSpursOnly(next);
      await fetch("/api/settings", {
          method: "PUT",
          headers: { "Content-Type": "application/json" },
          body: JSON.stringify({ spurs_filter_enabled: next }),
      });
  };

改善-3｜T-14（エンドツーエンド）のマトリクス注記が不明確
対象：test_design_v0.1.md セクション4（テスト観点マトリクス）
────────────────────────────────────────────────────────────────────────────────

【問題点】
テスト観点マトリクス（セクション4）のT-14は「単体: —」となっているが、
テスト設計書の本文ではT-14が総合テスト設計書（セクション3）に記載されており
整合性は取れている。しかし単体テスト設計書（セクション2）にT-14の項目が存在
しないにもかかわらず、マトリクスに「—」の行として存在しており、意図が不明確。

【変更案】
マトリクスのT-14行に以下の注記を追加する：

  T-14 | 全体 | —（総合テストにて代替） | ✅（シナリオ1〜6） | 高

改善-4｜ログファイルパスが logs/nba_news.log だがディレクトリ構成に logs/ が存在しない
対象：detail_design_v0.1.md セクション5-2（ロギング設定）
────────────────────────────────────────────────────────────────────────────────

【問題点】
セクション5-2のロギング設定で logs/nba_news.log をファイル出力先としているが、
基本設計書セクション2-2（ディレクトリ構成）に logs/ ディレクトリが定義されていない。

  - logs/ ディレクトリが存在しない状態でアプリを起動すると FileNotFoundError が発生する
  - 初回起動時の環境構築手順（setup.md）に mkdir logs が含まれているかも不明

【変更案】
ディレクトリ構成に logs/ を追加する。またはアプリ起動時に自動作成する処理を追加する：

  import os
  os.makedirs("logs", exist_ok=True)   # main.py の起動処理に追加

【🟢 軽微な指摘】
================================================================================

────────────────────────────────────────────────────────────────────────────────
軽微-1｜title_ja カラムのNULLABLE設計とDDLの不一致
対象：detail_design_v0.1.md セクション2-1（articlesテーブルDDL）
────────────────────────────────────────────────────────────────────────────────

DDLで title_ja TEXT と定義されており、NULLが許容されている。しかし
エラーハンドリング方針では「API上限時は title_ja=title_original（英語）で保存」
とあるため、title_ja はNULLにならないはず。設計意図を明確にするか、
DDLに DEFAULT '' または NOT NULL を追加することを検討する。


────────────────────────────────────────────────────────────────────────────────
軽微-2｜T-06（カテゴリ分類境界条件）が実Claude APIを使用する設計になっている
対象：test_design_v0.1.md T-06
────────────────────────────────────────────────────────────────────────────────

T-06の備考に「実際のClaude APIを呼び出す統合確認」とある。テスト環境では
「外部APIはモックで代替」（セクション1-3）としているが、T-06のみ実APIを呼び出す
設計になっており方針と矛盾する。実APIを使うテストはコスト発生・ネットワーク依存の
リスクがあるため、テスト方針との整合性を明記する必要がある。
