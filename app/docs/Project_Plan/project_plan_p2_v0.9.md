## 1. プロジェクト概要

### 1-1. フェーズ2の目的

フェーズ1で完成・本運用中のNBAニュース翻訳アプリに対し、以下を実施する。

1. **機能改修**：ユーザー要望（要望整理.txt）を反映した機能追加・変更
2. **プロセス改善**：フェーズ1の振り返りレビューで指摘された13項目の改善を開発手順に組み込む
3. **公開範囲拡大**：ローカル限定からスマホブラウザアクセス可能な環境への移行（認証付き）

### 1-2. フェーズ1からの継続事項

| 事項 | 内容 |
|---|---|
| 試合結果カテゴリ未対応 | Hoops Rumors非対応のため、フェーズ2で試合速報ソースを追加検討 |
| インジャリーレポートの境界定義 | F-03補足への明示をフェーズ2要件定義で対処 |
| ネタバレフィルタのON戻し | フェーズ1で未実装。フェーズ2で対応 |

### 1-3. 改修要件サマリー（要望整理.txt）

| # | 要望 | 区分 | 優先度 |
|---|---|---|---|
| 1 | カテゴリ再編（ドラフト・順位・試合日程・けが人・FA追加） | 機能追加 | 高 |
| 2 | ネタバレフィルタを試合結果・すべてタブのみに限定 | 機能変更 | 高 |
| 3 | タイトルクリックで記事全文翻訳を展開表示 | 機能追加 | 中 |
| 4 | ネタバレフィルタのOFF→ON戻しを可能に | バグ修正 | 高 |
| 5 | スマホブラウザからアクセス可能にする | インフラ変更 | 中 |
| 6 | 限られた人だけに見えるようにする | 法的対応・著作権 | 中 |
| 7 | 著作権上問題ないソースから取得・重複記事を1件表示 | 機能追加・法的対応 | 高 |

## 2. ガイドライン

| 区分 | 内容 |
|---|---|
| 法的制約 | スクレイピング禁止。RSSまたは公式APIのみ使用 |
| 著作権制約 | 記事全文の翻訳表示（要望3）は利用規約・著作権の確認後に実装判断 |
| コスト制約 | 全API（Claude API・試合日程API等）のコスト試算・上限設定を必須とする |
| インフラ変更の断絶（致命-2） | 要望5・6（スマホアクセス・認証）はフェーズ1の「ローカル限定・host="127.0.0.1"」というアーキテクチャを根本から変更する。基本設計（手順3）では「フェーズ1との差分設計」として独立したセクションを設け、移行方針を明示すること。他の機能改修（要望1〜4・7）とは独立したリスクとして扱う |
| 著作権・公開範囲制約 | 公開範囲は著作権法第30条「私的使用のための複製」の範囲（個人・家庭内・これに準ずる限られた範囲）に収めること。知人への公開はグレーゾーンのため手順2で法的リスクを確認する。アクセス制限（認証等）は著作権法上の私的利用範囲を担保するための手段として実装する |

| コスト制約 | 全API（Claude API・試合日程API・認証サービス等）のコスト試算・上限設定を必須とする |
| 著作権・公開範囲制約 | 公開範囲は著作権法第30条の私的利用範囲に収めること。認証機能は著作権上の公開範囲を担保するための手段として実装する（セキュリティ目的ではなく著作権法遵守が主目的） |
| 技術制約 | フェーズ1のコードベース（Python/FastAPI/React/SQLite）を継承する |
| 運用制約 | 公開する場合も著作権法第30条の私的利用範囲（個人・家族・これに準ずる限られた範囲）に限定する。商用利用・不特定多数への公開は禁止 |

## 7. リスクの事前認識

手順2（リスク事前調査）着手前の時点での暫定リスク認識。

| リスク | 内容 | 暫定対処方針 |
|---|---|---|
| 試合日程APIの有料化 | BALLDONTLIEの試合日程・スタッツが有料ティアのみの可能性 | 手順2で確認。有料の場合はスコープアウトを検討 |
| RSS全文翻訳の著作権 | 要望3の全文翻訳表示が引用範囲を超える可能性 | 手順2で確認。問題がある場合はリンクのみ表示に変更 |
| 公開インフラのコスト超過 | VPS・認証サービスのコストが想定外に高い可能性 | 手順2でコスト試算。月額1,000円以内を目安とする |
| 公開範囲が著作権法の私的利用範囲を超えるリスク | 知人への公開が著作権法第30条「これに準ずる限られた範囲」に該当しない可能性がある。認証で制限しても公開対象者の範囲次第では私的利用の範囲を超えると判断される可能性がある | 手順2で法的リスクを確認。家族・同居者への限定が最も安全。知人への公開は法的見解を確認の上で判断する |
| 認証の迂回リスク | 認証方式が脆弱で想定外の第三者がアクセスできる可能性（著作権法上の公開範囲逸脱につながる） | シンプルだが確実な認証方式を選定する（手順2で調査） |
| 重複記事排除の精度 | 類似記事の判定が難しく誤排除・排除漏れが発生する可能性 | 手順2で技術調査。シンプルなルールベース排除を優先 |
