## 5. 本表の使い方（次回以降の運用ルール）

1. 手順7のサブ手順生成を開始する前に、最新版のrequirements_p2_vX・basic_design_p2_vXの全変更点（機能要件表・カテゴリ表・DBスキーマ差分・フロントエンド変更点・インフラ変更点）を本表形式で列挙する。
2. 渡されたサブ手順リスト（7-A, 7-B, ...）の各指示文の実装内容と照合し、「対応サブ手順」列を埋める。
3. 空白または「未割当」になった項目があれば、サブ手順生成前にユーザーに報告し、追加サブ手順の必要性を確認する。
4. 全サブ手順が完了した後（手順7完了時）、本表を再度見直し、状態が❌・⚠️のままの項目がないことを最終確認する（7-I相当の最終レビューに本表チェックを追加する）。
5. **project_plan側でサブ手順番号がrenumberingされた場合（例：v1.7での7-J→7-J-2振り直し）、本表内の対応サブ手順列も同時に追従更新すること。本v1.1はこの運用ルール追加のきっかけとなった修正である（project_plan_p2_v1.7レビューYEL-01対応）。**
6. **運用判断によりサブ手順そのものを見送る（⏭️）場合がある（例：7-J-4）。この場合、本表では当該項目を「対応不要」または見送り理由付きで✅扱いとはせず、上位のbasic_design/requirements側の要件が別の手段で満たされていることを確認した上で記録すること。7-J-4の場合は、上位要件（DB永続化・本番データ蓄積）が本番の自走バッチで実質的に満たされているため、手順7完了の判定を妨げないものとした（v1.2・2026-06-27）。**
