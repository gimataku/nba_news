【🟡 要改善（1件）】
================================================================================

[YEL-01] §3技術リスク評価にDedup Processor（重複排除）の独立評価セクションが欠落
           かつ80%閾値の「根拠の実体」が評価書内に記載されていない

-----

【指摘内容】

§3（技術リスクの最終評価）は以下6セクションで構成されている：
  §3-1 外部公開・ホスティング
  §3-2 認証・セキュリティ
  §3-3 RSSソース安定性
  §3-4 Claude API
  §3-5 BALLDONTLIE API
  §3-6 SQLite / データ

フェーズ2の「新規追加コンポーネント」として唯一の専用モジュール（Dedup Processor）の
技術リスク評価セクションが存在しない。

Dedup Processorは：
・フェーズ1に存在しない完全新規コンポーネント
・基本設計レビューで RED-03・GRN-02 の指摘を受けた重要設計判断を含む
・80%閾値・7日間ウィンドウ・比較対象（英語原文）という3つの確定値がある
・精度リスク（誤排除・漏れ）がR-11として残存リスク受容されている

§6のR-11（重複記事判定精度不足）に「80%閾値は設計根拠あり」と記載されているが、
根拠の実体（「英語タイトルでの字句的類似度として誤検知と見逃しのバランスから選定」）が
評価書内に書かれておらず、「設計根拠あり」という参照形式に留まっている。

基本設計書§3-1には閾値根拠が記述されているため設計として確定済みだが、
リスク評価書を単独で読んだ場合に「なぜ80%か」が不明なまま残る。

-----

【修正案：§3に新セクション §3-7（または §3-2と§3-3の間）を追加】

§3-x. 重複排除（Dedup Processor）

  | リスク                   | 内容                                      | 対処状況 |
  |--------------------------|-------------------------------------------|---------|
  | 重複判定の誤排除（偽陽性） | 異なる内容の記事を重複と誤判定し非表示にする | ⚠️ 残存リスク（受容）。is_duplicate=Trueで保存のため後から修正可能 |
  | 重複の見逃し（偽陰性）    | 同一内容の記事が複数表示される              | ⚠️ 残存リスク（受容）。表示上の冗長のみで情報欠落なし |
  | 閾値の妥当性              | 80%の閾値が精度・再現率のバランスとして適切か | ⚠️ 残存リスク（受容）。根拠：英語タイトルの字句的類似度として誤検知と見逃しのバランスから選定（基本設計§3-1 GRN-02対応）。手順8の精度確認で調整余地あり |

  > ※ 比較対象：title（英語原文）・比較ウィンドウ：直近7日間（config.py定義）
  > ※ R-11として残存リスク一覧（§6）に詳細受容理由を記載済み

【🟢 軽微（1件）】
================================================================================

[GRN-01] §3-1技術リスクにフェーズ1との断絶
          （host="127.0.0.1"→"0.0.0.0"変更）リスク項目の明示がない

【現状】
§3-1（外部公開・ホスティング）のリスク表：
  - SQLiteデータ消失（Renderエフェメラル）→ ✅ 対処済み
  - Renderサーバーダウン → ⚠️ 残存リスク
  - デプロイ時のサービス停止 → ⚠️ 残存リスク

【指摘】
基本設計書§2が「フェーズ1との断絶（インフラ変更）」として独立セクションを設け、
FastAPI起動設定の変更（`host="127.0.0.1"` → `host="0.0.0.0"`）を明示していた。
フェーズ1リスク評価書v1.2 §1-2では「ローカルのみ（host="127.0.0.1"でFastAPI起動）」
が遵守事項として記録されていた。

この設計断絶そのものが§3-1の独立項目として記載されていない。
JWT認証（§3-2）により第三者アクセスリスクは対処済みだが、「フェーズ1では
本質的に外部からアクセス不可だったが、フェーズ2ではJWT認証がない場合に
誰でもAPIに到達できる状態になる」という変化の記録として追記が望ましい。

【修正案（§3-1テーブルへの追記）】

  | host変更による外部到達性 | フェーズ1はhost="127.0.0.1"で外部から到達不可だったが、フェーズ2はhost="0.0.0.0"で外部公開となる | ✅ JWT認証（§3-2）により第三者アクセスを制限。認証なしでは全エンドポイントにアクセス不可 |
