[RED-01] SECRET_KEY 未設定時の起動バリデーションが実装コードに存在しない
該当箇所：§2-2-1 auth/jwt.py

  auth/jwt.py の冒頭に以下のコードがある。
    SECRET_KEY = os.getenv("SECRET_KEY", "")   # 必須・未設定時は起動エラー

  コメントには「未設定時は起動エラー」と書かれているが、
  実際には空文字をデフォルト値として設定しているだけであり、
  起動エラーになる処理（例外 raise）が実装コードに含まれていない。

  python-jose の HS256 実装は空文字キーでもエラーなく動作するため、
  SECRET_KEY が未設定のまま起動した場合：
    - JWT の署名・検証は「空文字キー」で行われる
    - 攻撃者が同一ロジックで有効なトークンを生成可能（実質的に認証無効化）
    - Render の環境変数設定漏れが「サイレントに」本番認証を破壊する

  必要な修正：
    SECRET_KEY = os.getenv("SECRET_KEY", "")
    if not SECRET_KEY:
        raise RuntimeError("SECRET_KEY が未設定です。起動を中止します。")

  同様のバリデーションは §6 config.py にも必要。
  なお §2-2-2 users.py の USERNAME / USER_PASSWORD 未設定時の RuntimeError は
  正しく実装されており、同等の厳格さを jwt.py にも適用すること。

[YEL-01] config.py と auth/users.py の設定参照が二重管理になっている
該当箇所：§2-2-2 auth/users.py / §6 config.py

  §6 config.py には以下が定義されている。
    INIT_USERNAME      = os.getenv("USERNAME", "")
    INIT_USER_PASSWORD = os.getenv("USER_PASSWORD", "")

  しかし §2-2-2 auth/users.py の init_user() は
  config.py をインポートせず直接 os.getenv("USERNAME", "") を参照している。

  問題：環境変数名の変更や設定値の変更が生じた際、
  config.py と users.py の両方を修正する必要がある（管理漏れリスク）。

  修正方針：
    auth/users.py で `from config import INIT_USERNAME, INIT_USER_PASSWORD` を使う。
    config.py を唯一の設定ソースとして一元化する。

[YEL-02] CATEGORIES 配列に "schedule" を含める意図と用途が不明確
該当箇所：§5-2 hooks/useNews.js

  § 5-2 に以下の記述がある。
    const CATEGORIES = ["all", "trade_fa", "draft", "schedule", "injury", "column"];
    // ※ "schedule" は /api/schedule から取得（articlesとは別フック）

  問題：この CATEGORIES 配列がタブリストの定義として使われる場合は合理的だが、
  useNews フックの category パラメータ（/api/news?category=...）として渡される
  文脈でも使われた場合、/api/news?category=schedule という無意味なリクエストが
  バックエンドに飛ぶ。crud.get_articles(category="schedule") は 0 件を返すだけで
  エラーにならず、バグとして気づきにくい。

  修正方針：
    A案（推奨）: タブリスト用の配列（TAB_LIST）とAPIクエリ用の配列（API_CATEGORIES）を分ける。
      const TAB_LIST = ["all", "trade_fa", "draft", "schedule", "injury", "column"];
      const API_CATEGORIES = ["all", "trade_fa", "draft", "injury", "column"];
    B案: コメントに加えて guard 処理を追加する（schedule の場合は useNews を呼ばない）。
  どちらかを明示してコードに反映すること。

[YEL-03] §7 エラーハンドリング表に Authorization ヘッダーなしの HTTP 403 が未記載
該当箇所：§7 エラーハンドリング設計

  §7 の表には以下の記載がある。
    | JWT検証失敗（無効・期限切れ） | API Server | HTTP 401 を返す | ...

  FastAPI の HTTPBearer（auto_error=True デフォルト）は Authorization ヘッダー自体が
  ない場合、HTTP 403 Forbidden を返す（HTTP 401 ではない）。
  HTTP 401 は「トークンが存在するが無効」の場合のステータスコード。

  テスト設計書 TC-P2-02-1 では「HTTP 403 または 401」と正しく記述されているが、
  詳細設計書の §7 にはヘッダーなしのケースが未記載であり、設計と実装の乖離を生む。

  修正方針：§7 に以下の行を追加する。
    | Authorization ヘッダーなし | API Server | HTTP 403 を返す（FastAPI HTTPBearer の挙動） | フロントエンドでログイン画面にリダイレクト |

[YEL-04] D-06（JWTトークン保存先）の実装コードと注記が矛盾している
該当箇所：§5-1 hooks/useAuth.js

  §5-1 には useState によるメモリ保持の実装コードが書かれている（確定版の体裁）。
  同一箇所の D-06 注記には「実装前に確定が必要」と書かれている。

  矛盾：実装が先行しているのに「確定前」という注記が残っている。
  ページリロードでログアウトになるメモリ保持でよいのであれば、
  D-06 を「メモリ保持（useState）で確定」として §8 残課題から削除すべき。
  まだ確定していないなら実装コードを「仮実装」と明示すべき。

  修正方針：
    - メモリ保持で確定するなら D-06 を確定済みとして §8 から削除
    - sessionStorage 等への変更余地を残すなら「仮実装・D-06 確定後に更新」と注記を修正

  （セキュリティ補足：localStorage は XSS でトークンを読み取られるリスクがある。
  家族利用でも XSS リスクを完全に排除できない。
  メモリ保持＞sessionStorage＞localStorage の順でセキュリティ強度が高い。
  D-06 の選択はセキュリティ方針として設計書に理由を明記することを推奨する。）

[YEL-05] ログインエンドポイントにブルートフォース対策の記載がない
該当箇所：§3-3 POST /api/auth/login / §7 エラーハンドリング設計
セキュリティ観点：外部公開アプリにおける認証情報総当たりリスク

  POST /api/auth/login はレート制限・試行回数制限なし。
  Render で外部公開するため、悪意あるアクターが任意の頻度でログインを試行できる。
  HS256 形式の JWT の場合、パスワードを特定されるとトークン生成も可能になる。

  家族利用・個人アプリの文脈であれば許容する判断もありうるが、
  外部公開 URL にパスワード検証ロジックを公開することのリスクを
  設計書として明示し、対応方針を確定する必要がある。

  対応候補：
    A案（最低限）: ブルートフォース対策なし、として設計書に明示して受け入れる
    B案（推奨）: slowapi 等でログインエンドポイントに IP ベースのレート制限を追加
               （例: 1 IP あたり 5回/分）
  v0.1 で未対応のまま進むならリスク受け入れとして §8 または §7 に記載すること。

[GRN-04] 認証フロー（§5-1）に HTTPS 前提の明記がない
該当箇所：§5-1 hooks/useAuth.js / §3-3 POST /api/auth/login
セキュリティ観点

  パスワードを JSON ボディで POST 送信する設計は HTTPS 前提で成立する。
  基本設計書 §2-1 には「Render 自動プロビジョニング」の記載があるが、
  詳細設計書の認証フロー（§5-1）には HTTPS 前提の言及がない。
  「パスワードは HTTPS（TLS）経由でのみ送信される。HTTP での通信は
  Render により HTTPS にリダイレクトされる」等の注記を §5-1 または §3-3 に追記すること。
