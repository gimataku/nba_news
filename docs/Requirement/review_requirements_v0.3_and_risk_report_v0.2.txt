────────────────────────────────────────────────────────────────────────────────
致命-1｜ネタバレ防止機能（F-05）とBALLDONTLIE連携の設計が未定義のまま要件確定している
対象：requirements_v0.3.md
────────────────────────────────────────────────────────────────────────────────

【問題点】
要件定義書のF-05では「試合結果カテゴリの詳細スコア・スタッツをデフォルト非表示」
と定義されている。一方リスク調査書では「試合結果・スタッツはBALLDONTLIE APIから
取得する2系統設計」を引き継ぎ事項としている。

  - ニュースRSSから取得した記事本文にスコアが含まれている場合、それをClaudeが
    要約に含めるのか除外するのかが未定義
  - BALLDONTLIE（スタッツ）とRSSニュース記事（本文）の2系統があるとき、
    ネタバレ判定の「検出対象」はどちらなのか、または両方なのかが要件に未記載
  - この曖昧さはClaude APIプロンプト設計（手順4）に直接影響する

【変更案】
F-05に以下を追記する：

  - ネタバレ検出対象：RSSニュース本文中のスコア記述 および
                      BALLDONTLIE取得スタッツの両方
  - 要約生成時：「試合結果」カテゴリと判定された場合、Claudeはスコア・スタッツを
    要約文に含めない（プロンプトで制御）
  - スコア・スタッツの表示データソース：BALLDONTLIE API
    （RSSの本文スコアは使用しない）


────────────────────────────────────────────────────────────────────────────────
致命-2｜「カテゴリ自動分類」の4分類とネタバレ判定の関係が矛盾する可能性がある
対象：requirements_v0.3.md
────────────────────────────────────────────────────────────────────────────────

【問題点】
F-03でカテゴリは「トレード／契約／試合結果／コラム」の4分類とされている。
しかしNBAニュースには「試合結果」と「試合レビュー・分析コラム」が混在する。
試合後の選手インタビューや戦術分析記事は「試合結果」か「コラム」か判定が揺れる。

  - 「試合結果」カテゴリにのみネタバレ防止が適用される設計のため、
    分類精度が低いとネタバレが漏れる
  - 試合後コラムが「コラム」に分類された場合、スコアが本文に含まれていても
    ネタバレ防止が発動しない
  - 要件上この境界が未定義

【変更案】
カテゴリ定義に境界条件を追記する：

  - 「試合結果」：試合当日〜翌日の速報・ボックススコア記事
  - 「コラム」  ：試合分析・選手評価・戦術考察（スコア含む場合もあり）
  → 「コラム」カテゴリでもスコア・スタッツが含まれる場合は
    ネタバレ防止を適用するか否かをF-05に明記する


────────────────────────────────────────────────────────────────────────────────
致命-3｜リスク調査書のプロジェクト計画書（手順2）調査項目に未調査項目がある
対象：risk_report_v0.2.md
────────────────────────────────────────────────────────────────────────────────

【問題点】
プロジェクト計画書の手順2に明記されていた以下の調査項目が、
リスク調査書v0.2に対応する調査結果として存在しない。

  調査項目                                    リスク調査書v0.2の対応
  ─────────────────────────────────────────────────────────────────
  日本著作権法における「要約表示」の扱い      記載なし
  翻訳要約の表示が引用の範囲内かどうか        記載なし
  Claude APIの月次上限設定方法の確認          記載なし
                                              （上限額の記載はあるが設定手順の
                                               確認結果がない）
  RSSフィードに含まれる情報量（本文量）        記載なし
  チームフィルタに必要なメタ情報がRSSに        部分的
  含まれるか                                  （「チーム名タグなし」はESPNのみ。
                                               代替ソースは未確認）

日本著作権法の調査は、個人利用であっても「翻訳・要約の自動生成→表示」という
行為の合法性に直接関わる。設計着手の判断「進めてよい」はこの未調査を踏まえると
根拠が不完全。

【変更案】
以下のいずれかを取る：

  (A) 上記未調査項目をv0.3で補完してからリスク調査完了とする
  (B) 「日本著作権法については手順5（リスク正式評価）で対処する」と
      明示的にスコープ外宣言をリスク調査書に記載する

【🟡 要改善な問題】
================================================================================

────────────────────────────────────────────────────────────────────────────────
改善-1｜要件定義書のF-07（API使用上限管理）が実装観点で曖昧
対象：requirements_v0.3.md
────────────────────────────────────────────────────────────────────────────────

【問題点】
「月次コスト上限を設定、超過時は翻訳スキップ＋アラート表示」とあるが、
実装に必要な仕様が不足している。

  不足している定義：
  - 上限値はいくらか（リスク調査書では$20とあるが要件定義書に記載なし）
  - 「翻訳スキップ」時の表示はどうなるか（見出しのみ表示？記事非表示？）
  - アラートはどこに表示するか（ヘッダー？各記事？）
  - 月次リセットのタイミング（Claude APIの請求サイクルに合わせるか）

【変更案】
F-07の内容を以下のように具体化する：

  - 月次上限       ：$20（Anthropic Console上で設定）
  - 超過時の動作   ：翻訳・要約をスキップし、見出し（原文）のみ表示
  - アラート表示位置：ヘッダー部に「今月のAPI上限に達しました」バナー表示
  - リセット       ：毎月1日（Claude APIの請求サイクルに合わせる）


────────────────────────────────────────────────────────────────────────────────
改善-2｜Hoops RumorsのRSS実機確認が引き継ぎ事項に留まっており、リスクが残存している
対象：risk_report_v0.2.md
────────────────────────────────────────────────────────────────────────────────

【問題点】
引き継ぎ事項「各RSSを実機確認してチーム名タグの有無を確認」は、
設計着手前に完了していない。

  - Hoops Rumors・RealGMのRSSにSpurs関連の記事を特定できるメタ情報
    （タグ・カテゴリ）が含まれるかが未確認
  - 含まれない場合、チームフィルタ（F-04）はClaude APIによるチーム判定に
    全面依存することになる
  - Claude APIによる判定はコスト増加・レート上昇を招く可能性があり、
    設計前に確認すべき前提条件

【変更案】
基本設計（手順3）着手前に実機でRSSを取得し、以下を確認してリスク調査書に追記する：

  - 各RSSのフィールド一覧（title / description / category / tag 等）
  - Spurs関連記事に識別可能なメタ情報が含まれるか
  - 含まれない場合の代替手段（キーワードマッチ or Claude API判定）と
    コスト影響試算


────────────────────────────────────────────────────────────────────────────────
改善-3｜要件定義書のデータ保持要件が設計に対して曖昧
対象：requirements_v0.3.md
────────────────────────────────────────────────────────────────────────────────

【問題点】
非機能要件の「取得済みニュースはローカルDB（SQLite等）に保存。最新30日分」について、
削除の定義が不明確。

  - 「最新30日分」は取得日から30日か、記事掲載日から30日か不明
  - 30日を超えたデータを自動削除するバッチが必要か否かが未定義
  - SQLite「等」の「等」が曖昧。基本設計でDB選定が揺れる原因になる

【変更案】

  - 保持期間：記事取得日から30日
  - 削除方式：定時バッチ（翌日分取得時に30日超データを自動削除）
  - DB      ：SQLite（ローカル単一ファイル。変更する場合は基本設計で決定）


────────────────────────────────────────────────────────────────────────────────
改善-4｜リスク調査書のClaude API料金情報の参照先が古い可能性がある
対象：risk_report_v0.2.md
────────────────────────────────────────────────────────────────────────────────

【問題点】
Claude Haiku 4.5・Sonnet 4.6の料金として参照しているURLが
「platform.claude.com/docs/en/about-claude/pricing」となっており、
正規の料金ページ（anthropic.com/pricing）と異なる。

  - 参照URLが正規の料金ページか不明瞭
  - モデル名「claude-haiku-4-5」「claude-sonnet-4-6」は2026年5月時点での
    モデルであり、料金が変更されている可能性がある
  - コスト試算の前提が崩れると月次上限設定の根拠も変わる

【変更案】
製造着手前に https://www.anthropic.com/pricing で料金を再確認し、
リスク調査書の数値を更新する。コスト試算は設計フェーズで再度検証することを
引き継ぎ事項に追加する。

対象：risk_report_v0.2.md
────────────────────────────────────────────────────────────────────────────────

ヘッダーに「前版からの変更」が1行で書かれているが、v0.1の内容が添付されていない

参照ドキュメントとして明記することを推奨する。


────────────────────────────────────────────────────────────────────────────────
軽微-3｜「更新頻度」の表記が各ドキュメント間で統一されていない
対象：requirements_v0.3.md / risk_report_v0.2.md
────────────────────────────────────────────────────────────────────────────────

  要件定義書     ：「1日1〜2回（朝・夕の定時バッチ）」
  計画書         ：「1日2回取得」
  リスク調査書   ：「1日2回 × 20〜30件/回」（コスト試算の前提）

試算の根拠は「1日2回」で固定されているため、要件定義書の「1〜2回」という
幅のある記述は試算と不整合。どちらかに統一する。
