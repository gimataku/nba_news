## 1. プロジェクト概要

### 1-1. フェーズ2の目的

フェーズ1で完成・本運用中のNBAニュース翻訳アプリに対し、以下を実施する。

1. **機能改修**：ユーザー要望（要望整理.txt）を反映した機能追加・変更
2. **プロセス改善**：フェーズ1の振り返りレビューで指摘された13項目の改善を開発手順に組み込む
3. **公開範囲拡大**：ローカル限定からスマホブラウザアクセス可能な環境への移行（認証付き）

### 1-2. フェーズ1からの継続事項

| 事項 | 内容 |
|---|---|
| 試合結果カテゴリ未対応 | Hoops Rumors非対応のため、フェーズ2で試合速報ソースを追加検討 |
| インジャリーレポートの境界定義 | F-03補足への明示をフェーズ2要件定義で対処 |
| ネタバレフィルタのON戻し | フェーズ1で未実装。フェーズ2で対応 |

### 1-3. 改修要件サマリー（要望整理.txt）

| # | 要望 | 区分 | 優先度 |
|---|---|---|---|
| 1 | カテゴリ再編（ドラフト・順位・試合日程・けが人・FA追加） | 機能追加 | 高 |
| 2 | ネタバレフィルタを試合結果・すべてタブのみに限定 | 機能変更 | 高 |
| 3 | タイトルクリックで記事前文翻訳を展開表示 | 機能追加 | 中 |
| 4 | ネタバレフィルタのOFF→ON戻しを可能に | バグ修正 | 高 |
| 5 | スマホブラウザからアクセス可能にする | インフラ変更 | 中 |
| 6 | 限られた人だけに見えるようにする | セキュリティ追加 | 中 |
| 7 | 著作権上問題ないソースから取得・重複記事を1件表示 | 機能追加・法的対応 | 高 |

### 手順2：リスク事前調査 ⬜ 🔧

- **アウトプット**: リスク調査書（フェーズ2版）
- **成果物**: `risk_report_p2_v0.1.md`（作成予定）
- **フェーズ1からの改善点**:
  - ライブラリ選定時にタイムアウト・エラーハンドリング仕様を必ず確認する（レビュー提言9）
  - 「使えないことの確認」（404・認証エラー等）を設計前に完了させる

#### 法的調査

- [ ] **試合日程・スタッツAPIの利用規約確認**
  - 候補：BALLDONTLIE（gamesエンドポイント）・ESPN API・NBA Stats API等
  - 無料ティアで試合日程・スタッツが取得できるか確認
  - 「AIなし・他サイト」（要望1）の意味を確認：外部サイトURLリンクか、APIデータ取得か
- [ ] **追加RSSソースの利用規約確認**
  - 著作権上問題ないソースの特定（要望7）
  - 現行ソース（Hoops Rumors・Hoops Wire・The Cold Wire NBA）の利用規約再確認
- [ ] **RSS記事前文の翻訳表示の著作権確認**（要望3）
  - 前文（description/summary）の翻訳表示が引用の範囲内か確認
  - 元記事リンク表示を条件とした場合の合法性確認
- [ ] **公開方式の法的確認**（要望5・6）
  - VPS/クラウド（Render・Railway等）でのホスティング費用・規約確認
  - 個人利用の認証付き公開が著作権上問題ないか確認

#### コスト調査

- [ ] **試合日程・スタッツAPIのコスト試算**
  - BALLDONTLIE無料ティアの制限確認（リクエスト数・エンドポイント）
  - 有料APIが必要な場合のコスト試算
- [ ] **公開インフラのコスト試算**（要望5・6）
  - VPS/クラウド費用の試算（月額）
  - 認証サービス（Auth0等）のコスト確認
- [ ] **Claude APIコストの再試算**
  - カテゴリ増加・前文翻訳追加によるトークン数増加の試算
  - 月次上限の見直し検討

#### 技術調査

- [ ] **試合日程・スタッツAPIの実機確認**
  - BALLDONTLIEのgamesエンドポイントで試合日程・会場・スタッツが取得できるか確認
  - 取得できない場合の代替手段を確認
- [ ] **追加RSSソースのRSS実機確認**
  - フィードURL・HTTPステータス・`<category>`タグの存在確認
  - Spursフィルタ（categoryタグ）が機能するか確認
- [ ] **重複記事排除の技術調査**（要望7）
  - 同一記事の判定基準（タイトル類似度・URLドメイン・pubDate等）
  - 実装方式の検討（SQLite内での重複チェック・LLMによる類似判定）
- [ ] **認証方式の技術調査**（要望6）
  - シンプルなパスワード認証・Basic認証・OAuth等の選定
  - FastAPIへの組み込み方法の確認
