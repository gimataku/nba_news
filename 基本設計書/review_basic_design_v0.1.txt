【🔴 致命的な問題】
================================================================================

────────────────────────────────────────────────────────────────────────────────
致命-1｜BALLDONTLIE Score Fetcherの取得タイミングが未定義で、
        ネタバレ防止が機能しないケースがある
対象：セクション3-1（データフロー）
────────────────────────────────────────────────────────────────────────────────

【問題点】
データフロー（3-1）では「category=試合結果 または has_score=True の記事のみ
Score Fetcherを実行する」とある。しかし Score Fetcher でBALLDONTLIEから
試合データを取得する際、どの試合IDを使って照会するかの設計がない。

  - RSSの記事本文にgame_idは含まれない
  - 記事のpubDateとチームID（Spurs=27）から「その日のSpurs試合」を特定する
    必要があるが、この照合ロジックが未設計
  - 照合できなかった場合（例：記事日付とBALLDONTLIE上の試合日がずれる）、
    score_dataがNULLになるが、その場合の画面表示が未定義
  - ネタバレ防止はONなのにスコアが表示できない（NULLのまま）状態が発生しうる

【変更案】
データフローのScore Fetcherステップに以下を追記する：

  [Score Fetcher]
  1. 記事のpubDateから対象日（当日 or 前日）を決定
  2. BALLDONTLIE /v1/games?team_ids[]=27&start_date=YYYY-MM-DD&end_date=YYYY-MM-DD
     でSpursの試合を検索
  3. 試合が見つかった場合 → game_id を取得し score_data に保存
  4. 試合が見つからなかった場合 → score_data = NULL
     → フロントエンドは「スコアを表示」ボタンを非表示にする

致命-2｜API上限超過の判定ロジックに欠陥がある
対象：セクション7-3（API上限超過時の動作）
────────────────────────────────────────────────────────────────────────────────

【問題点】
セクション7-3では「Claude APIから429エラーが返った場合、
app_settings.api_limit_exceeded = "true" に更新する」とある。
しかし月次リセット（毎月1日）のリセット処理が設計されていない。

  - 月次上限超過で api_limit_exceeded = "true" になった後、翌月1日になっても
    自動でリセットされる設計がない
  - 手動で設定を戻さない限り、永久に翻訳スキップ状態が続く
  - 要件定義書（F-07補足）に「月次リセット：毎月1日」と明記されているが
    設計に未反映

【変更案】
バッチ処理フローの冒頭またはSchedulerに以下を追加する：

  [Scheduler / バッチ起動時]
  → 現在日付が前回リセット日と異なる月の場合：
     app_settings.api_limit_exceeded = "false" にリセット
     app_settings.api_reset_month を当月に更新

  app_settings テーブルに api_reset_month（例："2026-05"）カラムを追加し、
  月が変わったかどうかを判定する。

【🟡 要改善な問題】
================================================================================

────────────────────────────────────────────────────────────────────────────────
改善-1｜HoopsHype RSSの<category>タグによるSpursフィルタが未検討
対象：セクション3-1（データフロー：Spursチェック）
────────────────────────────────────────────────────────────────────────────────

【問題点】
データフロー（3-1）のSpursチェックは「<category>タグのキーワードマッチ」と
しているが、これはHoops Rumorsで実機確認済みの仕様。フェールオーバー時に
使用するHoopsHypeの<category>タグ構成は実機未確認。

  - HoopsHypeがフェールオーバー元になった際、<category>タグがなければ
    Spursフィルタが機能しない
  - risk_report_v0.5 の引き継ぎ事項にHoopsHypeの実機確認は含まれていないため、
    設計上の穴になっている
  - is_spurs が常にFalseになり、「チームのみ」フィルタで記事が全件消える可能性がある

【変更案】
フィルタ設計に以下のフォールバックロジックを追記する：

  Spursチェックの優先順位:
  1. <category>タグにキーワードが含まれる → is_spurs = True
  2. <category>タグが存在しない または マッチなし
     → <title> と <description> にキーワードマッチを試みる
  3. いずれにもマッチなし → is_spurs = False

  またHoopsHypeのRSS実機確認を詳細設計着手前の必須事項として追加することを推奨。


────────────────────────────────────────────────────────────────────────────────
改善-2｜GET /api/articles/{id}/score エンドポイントが不要になっている可能性がある
対象：セクション6-1（APIエンドポイント一覧）
────────────────────────────────────────────────────────────────────────────────

【問題点】
バッチ処理フローでは「Score Fetcherで取得したスコアを score_data カラムに
保存する」設計になっている。つまりスコアはバッチ時点でDBに保存済みであり、
フロントエンドはスコア表示時に GET /api/articles のレスポンスに含まれる
score_data をそのまま使えばよい。

  - GET /api/articles/{id}/score を別途呼び出す設計にすると、「スコアを表示」
    ボタン押下時に追加APIリクエストが発生する
  - DBに保存済みのデータを再取得するだけであれば、GET /api/articles の
    レスポンスに score_data を含めるだけで足りる
  - ただし score_data はネタバレ情報のため、GET /api/articles のレスポンスに
    常時含めることの是非を設計で判断する必要がある

【変更案】
以下の2つの設計方針のどちらかを選択し明記する：

  方針A（シンプル）: score_dataをGET /api/articlesのレスポンスに常時含める
    → フロントエンドはボタン押下時にローカルのscore_dataを表示するだけ
    → /api/articles/{id}/score は不要

  方針B（セキュア）: score_dataはGET /api/articles/{id}/scoreでのみ返す
    → ボタン押下時に追加リクエストが発生
    → ネタバレ情報の分離が明確

  ローカル限定アプリのため方針Aで十分と思われるが、意図を明示することを推奨。
