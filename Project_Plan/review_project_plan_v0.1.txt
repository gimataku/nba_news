【🔴 致命的な問題】
================================================================================

────────────────────────────────────────────────────────────────────────────────
致命-1｜手順7（テスト設計書作成）の位置が根本的におかしい
────────────────────────────────────────────────────────────────────────────────

【問題点】
手順7がテスト設計であるにもかかわらず、手順6（製造）の後に置かれている。
備考に「詳細設計（手順4）と並走」と書かれているが、手順の番号体系と矛盾している。

  - 手順の番号は実行順序を示すと読み手は解釈する
  - 「手順7は手順4と並走」という記述は、番号順に読むと意味が取れない
  - 製造着手時にテスト設計が未完成になるリスクがある

【変更案】
以下のように手順を再編成する：

  手順4：詳細設計
  手順5：テスト設計書作成（手順4と並走）※番号は便宜上
  手順6：リスク正式評価
  手順7：製造
  手順8：単体テスト
  手順9：総合テスト

または、手順4の中に「テスト設計書の並走着手」を明示的に組み込み、
独立手順としての手順7を削除する。


────────────────────────────────────────────────────────────────────────────────
致命-2｜スケジュール・期間が一切存在しない
────────────────────────────────────────────────────────────────────────────────

【問題点】
全11手順にわたって、日程・期間・マイルストーンの記載がゼロ。
手順10のトライアル運用のみ「2〜4週間」という記載があるが、全体の終了見込みが不明。

  - いつ本運用を開始できるかが誰にも分からない（作者自身も含め）
  - フェーズ2（NFL拡張）の設計見直し時期を「手順11で設定」とあるが、
    そもそも手順11がいつになるか不明
  - 手順2（リスク調査）が「進行中」だが、いつ終わるかの目安がない

【変更案】
各手順に「目標期間」または「目標完了日」欄を追加する。
個人プロジェクトであっても、期間見積もりは手戻りの抑止力になる。


────────────────────────────────────────────────────────────────────────────────
致命-3｜トライアル運用（手順10）に成功基準・中止条件がない
────────────────────────────────────────────────────────────────────────────────

【問題点】
確認項目はチェックリスト形式で列挙されているが、「何をもって合格とするか」が未定義。

  - 「翻訳・要約の品質確認」→ どのレベルなら本運用に移行するのか不明
  - 「コスト実績の確認」→ 試算と何割乖離したら中止・再設計するのかの基準がない
  - 基準なしのトライアルは、実質的に「様子を見る」だけになる

【変更案】
以下のような移行基準・中止基準を追加する：

  ＜移行基準（例）＞
  - Claude APIコストが月次試算の150%以内
  - 翻訳品質：読了に支障がないレベル（主観評価でも基準を言語化する）
  - カテゴリ分類精度：目視確認で80%以上の正確性
  - バッチ成功率：2週間で95%以上

  ＜中止・再設計基準（例）＞
  - コストが試算の200%を超えた場合
  - RSSフィードが安定取得できないソースが主要ソースの過半を占める場合

【🟡 要改善な問題】
================================================================================

────────────────────────────────────────────────────────────────────────────────
改善-1｜手順2のリスク調査項目に抜けがある
────────────────────────────────────────────────────────────────────────────────

【問題点】
技術調査の項目が浅い。

  - ESPN の RSS フィードは2023年頃に廃止・縮小されたという報告が複数ある。
    「提供状況の確認」とあるが、代替手段（公式API、他ソース）が見つからなかった
    場合の代替案が計画に存在しない
  - ローカル環境でのスケジューラ（cron等）の実現方式確認が抜けている
    （PCがスリープしていたらバッチが動かない等）
  - Claude API のレートリミットが記事の一括処理に影響しないかの確認が抜けている

【変更案】
技術調査に以下の項目を追加する：

  - [ ] ESPN RSSが利用不可だった場合の代替ニュースソースの特定
  - [ ] ローカル環境でのバッチスケジューリング方式の確認
        （cron / タスクスケジューラ / 常駐プロセス）
  - [ ] Claude API レートリミットと記事処理数の整合性確認


────────────────────────────────────────────────────────────────────────────────
改善-2｜成果物一覧（セクション4）と手順の対応に漏れがある
────────────────────────────────────────────────────────────────────────────────

【問題点】
  - 手順7（テスト設計書作成）の対応手順欄が「手順4・7」と書かれており、混乱を招く
  - 手順6（製造）に対応する成果物（ソースコード・環境構築手順書等）が
    成果物一覧に存在しない

【変更案】
成果物一覧に以下を追加する：

  | ソースコード一式   | /src/       | 手順6 | ⬜ 未着手 |
  | 環境構築手順書     | setup.md    | 手順6 | ⬜ 未着手 |

個人利用であっても、環境が再現できなくなるリスクへの備えとして有効。


────────────────────────────────────────────────────────────────────────────────
改善-3｜「お気に入りチーム」がプロジェクト概要表に記載されている
────────────────────────────────────────────────────────────────────────────────

【問題点】
プロジェクト概要のメタ情報テーブルに「お気に入りチーム: San Antonio Spurs」が
含まれているが、これは要件であって計画書のヘッダメタ情報ではない。

【変更案】
プロジェクト概要表から削除し、セクション5（制約・前提条件）または
要件定義書（requirements_v0.3.md）に移動する。
計画書のメタ情報は「誰が・いつ・何を作るか」に絞るべきである。


────────────────────────────────────────────────────────────────────────────────
改善-4｜フェーズ2（NFL拡張）の扱いが曖昧
────────────────────────────────────────────────────────────────────────────────

【問題点】
「本計画書はフェーズ1のみを対象とする」と明記されているのは良い。
しかし手順11に「フェーズ2の設計見直し時期の設定」が含まれており、
フェーズ1の計画書にフェーズ2のアクションが混入している。

【変更案】
手順11の該当項目を削除し、フェーズ2は別計画書
（project_plan_phase2_v0.1.md）で管理することを計画書に明記する。

────────────────────────────────────────────────────────────────────────────────
軽微-2｜凡例テーブルの位置
────────────────────────────────────────────────────────────────────────────────

凡例（✅🔄⬜）が手順3の直前に置かれており、手順1・2を読んだ後で定義が出てくる。
ドキュメント冒頭（セクション3の先頭）に移動することで可読性が上がる。
現状でも手順1・2には記号が使われているため、前後関係が逆転している。

────────────────────────────────────────────────────────────────────────────────

現時点ではv0.1のみなので問題ないが、今後レビュー指摘を反映した際に
バージョンアップのルール（いつマイナーを上げるか）を決めておくと管理しやすい。
